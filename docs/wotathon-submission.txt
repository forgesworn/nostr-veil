The Problem
-----------

NIP-85 got the hard part right: a standard format for verifiable trust
assertions on Nostr. That foundation is what makes everything below
possible.

The open question is what happens when contributor privacy matters.
Abuse reporting, whistleblowing, anonymous peer review. In those
contexts, the person scoring needs to stay hidden. Not because NIP-85
is wrong, but because the threat model is different.


What nostr-veil Does
--------------------

nostr-veil is a privacy extension for NIP-85. It layers ring signature
anonymity onto standard NIP-85 events.

A trust circle of N members can collectively score a subject. Each member
contributes anonymously via LSAG (Linkable Spontaneous Anonymous Group)
signatures. The result is a standard kind 30382 event that any existing
Nostr client reads unchanged. Three extra tags carry the proof:

  veil-ring       Sorted public keys of circle members
  veil-threshold  How many signed / circle size (e.g. "5", "8")
  veil-sig        One per signer: LSAG signature + key image

Key images prevent double-signing. If the same person tries to contribute
twice, the duplicate key image is detected. You get anonymity AND ballot
integrity.

NIP-85 is permissive about tags. Clients that don't recognise the veil
tags simply ignore them and display the metrics as normal. No fork, no new
event kind, no breaking change. Backwards compatible by design.


The Trust Trilemma
------------------

NIP-85 nails verifiability and portability. nostr-veil adds the third
leg: contributor privacy, without sacrificing either of the first two.

  Property     NIP-85       NIP-85 + nostr-veil
  ----------   ------       -------------------
  Verifiable    Yes             Yes
  Private        —              Yes
  Portable      Yes             Yes

The assertions remain standard NIP-85 events any client can consume.
Veil-aware clients can additionally verify the ring proofs.


What Makes This Different
-------------------------

LSAG ring signatures are battle-tested in Monero since 2014, but there
was no JavaScript implementation targeting the Nostr ecosystem.
@forgesworn/ring-sig implements the maths from scratch on secp256k1.

This is not hackathon code. The cryptographic primitives are independently
published npm packages with their own test suites and release cycles.
nostr-veil is a thin integration layer over:

  @forgesworn/ring-sig   SAG + LSAG on secp256k1, independently published
  @noble/curves          Audited elliptic curve library by Paul Miller
  @noble/hashes          Audited hash primitives

nostr-veil itself has 118 tests across 11 files. Full NIP-85 kind coverage
(30382-30385 + 10040) with builders, parsers, and relay filters for all
five kinds, plus validators for the four assertion kinds.


Architecture
------------

Two subpath exports, each usable independently:

  nostr-veil/nip85   NIP-85 foundation. Builders, parsers, validators,
                     and relay filters for kinds 30382-30385 and 10040.
                     Standalone -- no dependency on the proof layer.

  nostr-veil/proof   Ring-signature proof layer. Trust circles, anonymous
                     contributions via LSAG, aggregation (default: median),
                     and verification. Depends on nip85 for event types.

Supporting:
  src/signing.ts     Schnorr event signing (NIP-01 serialisation)


NIP-85 Interoperability
-----------------------

Full coverage of all NIP-85 event kinds:

  Kind 30382  User assertions (rank, followers, zap activity)
  Kind 30383  Event assertions
  Kind 30384  Addressable event assertions
  Kind 30385  Identifier assertions (with k-tag)
  Kind 10040  Provider declarations (plaintext + NIP-44 encrypted)

All 17 user assertion metric tags supported. The output of
aggregateContributions() is a standard NIP-85 event. Any NIP-85 client
should display it correctly, since the veil tags are additive and the
base event structure is unchanged.


How It Works
------------

1. Create a trust circle from a set of member pubkeys.
2. Each member calls contributeAssertion() independently. Under the hood,
   this calls lsagSign from @forgesworn/ring-sig, producing an LSAG
   signature over a canonical message { circleId, subject, metrics }.
3. aggregateContributions() collects the signed contributions, validates
   every LSAG signature, computes the median of each metric, and outputs
   a standard NIP-85 kind 30382 event with the three veil tags.
4. verifyProof() takes the published event and:
   - Reconstructs each LSAG signature against the ring
   - Confirms each signature is valid
   - Checks key images are distinct (no double-signing)
   - Confirms the count of valid signers meets the threshold

At no point does verification require knowing which member produced which
signature. The ring is public. The identities of the actual signers are not.


Selective Disclosure (via nsec-tree)
-------------------------------------

nostr-veil provides the anonymity. The companion library nsec-tree
provides optional selective disclosure on top. Together they enable a
circle member to prove they contributed without revealing anything about
the other signers.

nsec-tree derives anonymous and public personas from a single master key.
A linkage proof connects them cryptographically. Two modes:

  Blind proof   "I was in the circle" -- no derivation path revealed.
  Full proof    Reveals the derivation path -- for legal proceedings or
                formal attribution.

This is demonstrated in the interactive demo but is not part of the
nostr-veil library itself.

What This Enables
-----------------

Before nostr-veil, anonymous trust contributions on Nostr were impossible.
After it, they are an npm install away.

Abuse reporters can vouch without exposure. Journalists can verify sources
without creating a traceable graph. Trust circles can form, contribute, and
dissolve without any single member's participation being attributable.

The people who need Web of Trust most can finally use it.
