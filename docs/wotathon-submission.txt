The Problem
-----------

Nostr has a standard for publishing trust scores -- NIP-85 (Trusted
Assertions). A provider can publish reputation data about a user
(follower count, rank, engagement metrics) in a format any Nostr app
can read. That foundation is what makes everything below possible.

The open question is what happens when contributor privacy matters.
Abuse reporting, whistleblowing, anonymous peer review. In those
contexts, the person giving the score needs to stay hidden. Not
because NIP-85 is wrong, but because the threat model is different.


What nostr-veil Does
--------------------

nostr-veil is a privacy extension for NIP-85. It layers ring-signature
anonymity onto standard trust score events.

A ring signature is a cryptographic technique where someone signs a
message on behalf of a group. A verifier can confirm "someone in this
group signed this" but cannot tell who. Think of it like a sealed
ballot -- you can count the votes but not trace them.

A trust circle of N members can collectively score a subject. Each
member contributes anonymously using ring signatures. The result
is a standard NIP-85 event that any existing Nostr app reads unchanged.
Three extra tags carry the proof:

  veil-ring       The group of members who could have contributed
  veil-threshold  How many contributed / circle size (e.g. "5", "8")
  veil-sig        One per contributor: ring signature + duplicate token

The duplicate token (called a "key image") prevents double-signing. If
the same person tries to contribute twice, the duplicate is detected.
You get anonymity AND ballot integrity.

NIP-85 is permissive about tags. Apps that don't recognise the veil
tags simply ignore them and display the scores as normal. No fork, no
new event kind, no breaking change. Backwards compatible by design.


The Trust Trilemma
------------------

NIP-85 nails verifiability and portability. nostr-veil adds the third
leg: contributor privacy, without sacrificing either of the first two.

  Property     Standard NIP-85    With nostr-veil
  ----------   ---------------    ---------------
  Verifiable    Yes                Yes
  Private        —                 Yes
  Portable      Yes                Yes

The scores remain standard NIP-85 events any Nostr app can display.
Privacy-aware apps can additionally verify the cryptographic proofs.


What Makes This Different
-------------------------

Ring signatures are battle-tested in production (Monero has used them
since 2014), but there was no JavaScript implementation targeting the
Nostr ecosystem. @forgesworn/ring-sig implements the cryptography from
scratch on the same elliptic curve Bitcoin and Nostr use (secp256k1).

This is not hackathon code. The cryptographic primitives are independently
published npm packages with their own test suites and release cycles.
nostr-veil is a thin integration layer over:

  @forgesworn/ring-sig   Ring signature library, independently published
  @noble/curves          Audited elliptic curve library by Paul Miller
  @noble/hashes          Audited hash primitives

nostr-veil itself has 118 tests across 11 files. Full NIP-85 kind coverage
(30382-30385 + 10040) with builders, parsers, and relay filters for all
five kinds, plus validators for the four assertion kinds.


Architecture
------------

Two subpath exports, each usable independently:

  nostr-veil/nip85   NIP-85 foundation. Builders, parsers, validators,
                     and relay filters for kinds 30382-30385 and 10040.
                     Standalone -- no dependency on the proof layer.

  nostr-veil/proof   Ring-signature proof layer. Trust circles, anonymous
                     anonymous contributions, aggregation (default: median),
                     and verification. Depends on nip85 for event types.

Supporting:
  src/signing.ts     Schnorr event signing (NIP-01 serialisation)


NIP-85 Interoperability
-----------------------

Full coverage of all NIP-85 event kinds:

  Kind 30382  User assertions (rank, followers, zap activity)
  Kind 30383  Event assertions
  Kind 30384  Addressable event assertions
  Kind 30385  Identifier assertions (with k-tag)
  Kind 10040  Provider declarations (plaintext + NIP-44 encrypted)

All 17 user assertion metric tags supported. The output of
aggregateContributions() is a standard NIP-85 event. Any NIP-85 client
should display it correctly, since the veil tags are additive and the
base event structure is unchanged.


How It Works
------------

1. Create a trust circle from a set of member public keys.
2. Each member submits their scores independently. Under the hood,
   each submission is wrapped in a ring signature -- proving group
   membership without revealing identity.
3. The contributions are collected, every signature is validated, and
   the median of each metric is computed. The output is a standard
   NIP-85 trust score event with three extra tags carrying the proofs.
4. Verification checks:
   - Each ring signature is valid (a real member signed it)
   - Duplicate-detection tokens are all different (nobody voted twice)
   - Enough members contributed to meet the threshold

At no point does verification require knowing which member produced
which score. The group membership is public. The identities of the
actual contributors are not.


Selective Disclosure (via nsec-tree)
-------------------------------------

nostr-veil provides the anonymity. The companion library nsec-tree
provides optional selective disclosure on top. Together they enable a
circle member to prove they contributed without revealing anything about
the other signers.

nsec-tree generates separate anonymous identities from a single master
key. A linkage proof connects them cryptographically. Two modes:

  Blind proof   "I was in the circle" -- no further details revealed.
  Full proof    Reveals the full identity chain -- for legal proceedings
                or formal attribution.

This is demonstrated in the interactive demo but is not part of the
nostr-veil library itself.


Companion Libraries
-------------------

These complement nostr-veil but are independently maintained:

  nsec-tree      Generate separate anonymous identities from one master key
  canary-kit     Detect when someone is being coerced (duress signals)
  signet         Decentralised identity verification for Nostr

Interactive Demo
----------------

Seven-screen whistleblower scenario demonstrating the full flow:

  0. Setup          Identity derivation via nsec-tree
  1. The Circle     Form a trust circle of 8 journalists
  2. The Source     Score an anonymous whistleblower's credibility
  3. The Veil       Watch ring signatures being constructed
  4. Verification   5-step verification pipeline (real cryptography)
  5. The Reveal     Optional selective disclosure
  6. The Recap      Trust graph with ring endorsement + duress animations

What This Enables
-----------------

Before nostr-veil, anonymous trust contributions on Nostr were impossible.
After it, they are an npm install away.

Abuse reporters can vouch without exposure. Journalists can verify sources
without creating a traceable graph. Trust circles can form, contribute, and
dissolve without any single member's participation being attributable.

The people who need reputation systems most can finally use them safely.
