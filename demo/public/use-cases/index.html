Use cases developers can implement.

nostr-veil turns NIP-85 trust assertions into anonymous, threshold-backed proofs. A verifier can check that distinct members of a public circle contributed, without learning which members took part.

Proof contract

Be precise about what the proof gives you.

The strongest use cases are honest at the boundary. nostr-veil handles the anonymous threshold proof; the application supplies evidence, policy, canonicalisation, expiry, and operational safety.

nostr-veil proves

  • Each contribution was signed by some member of the public circle.
  • Distinct contributors are counted once for the same circle, subject, and scope.
  • The published metric tags match the signed contribution messages.
  • Proof v2 binds the proof to the assertion kind and subject hint.
  • The output remains an ordinary NIP-85 event on ordinary relays.

Your application supplies

  • Circle admission, rotation, Sybil policy, and governance.
  • Evidence workflows, appeals, reviewer methodology, and disputes.
  • Identifier canonicalisation for domains, packages, relays, and vendors.
  • Expiry, revocation, freshness, and incident response.
  • Transport privacy when timing, IP address, or relay metadata matters.
NIP-85 shape

Pick the subject first. The helper follows.

Most implementation mistakes come from scoring the wrong subject. Start with the thing a verifier will act on, then choose the matching assertion kind.

30382

User pubkey

Use aggregateContributions when the subject is a Nostr account.

30383

Event id

Use aggregateEventContributions for claims, proposals, announcements, and reports.

30384

Addressable event

Use aggregateAddressableContributions for long-form notes, lists, and NIP-33 records.

30385

External identifier

Use aggregateIdentifierContributions for packages, domains, relays, vendors, and services.

Minimal flow

Four calls for a v2 proof.

Use v2 for new workflows where kind and subject binding matter. Legacy v1 events still verify unless your policy explicitly rejects them. 

import {
  createTrustCircle,
  contributeAssertion,
  aggregateContributions,
  validateAssertionStrict,
  verifyProof,
} from 'nostr-veil'

const proofVersion = 'v2'
const circle = createTrustCircle(reviewers.map(r => r.pubkey))

const contributions = reviewers.map(reviewer =>
  contributeAssertion(
    circle,
    subjectPubkey,
    { rank: reviewer.rank },
    reviewer.privateKey,
    circle.members.indexOf(reviewer.pubkey),
    { proofVersion },
  ),
)

const assertion = aggregateContributions(circle, subjectPubkey, contributions, { proofVersion })
const syntax = validateAssertionStrict(assertion)
const proof = verifyProof(assertion, { requireProofVersion: 'v2' })

if (!syntax.valid || !proof.valid) {
  throw new Error([...syntax.errors, ...proof.errors].join('; '))
}

Roadmap boundary

What is shipped, and what still needs a profile.

The core library ships anonymous circles, threshold aggregation, strict validation, proof v2 binding, and scoped federation. Some high-value workflows need additional event formats or handshakes.

planned

Anonymous endorsements

A circle co-signs a credential or attestation, not only a numeric score. The profile needs holder binding, expiry, revocation, and presentation rules.

planned

Anonymous gated access

A relay or community admits a vouched member through a challenge/response flow without learning which circle member opened the door.

planned

Revocation

A contributor or circle can withdraw a vouch or mark a score stale while preserving the anonymity properties of the original proof.

planned

Time-bounded trust

Provider profiles can make expiry first-class so reputation reflects current trust rather than old participation.
